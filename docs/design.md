# 设计思路

## 核心理念

**大模型做慢思考、经典算法做快反应。** 分层的本质,是让「决策的抽象层级」和
「响应的时间尺度」对齐:层级越高越抽象、调用越稀疏、可用越大的模型;层级越低
越具体、频率越高、必须用越快的模型。**大模型只做规划、不进控制环**,运动控制与
避障仍由成熟的经典算法执行。

场景图(Scene Graph)贯穿全系统:把连续传感数据压缩成「节点=物体/区域/平台,
边=空间/语义关系」的结构化表示,让 Qwen 能"读懂"物理世界,也让规划结果可被
校验与可视化。

## 三层架构

### L1 顶层 · 全局任务规划器(Qwen3.6-27B,中心站)
输入:自然语言任务、全局粗粒度场景图、集群资源清单、硬约束(禁飞区/时间窗/通信)。
处理:任务理解与分解(模糊指令→子任务 DAG)、集群编组(按任务/地理/能力分 K 组)、
组任务分配、全局时序与资源协调。输出:结构化的「分组—任务」方案。

### L2 中层 · 分组协调器(Qwen3.6-27B,边缘服务器/长机,每组一个实例)
输入:本组目标、局部场景图、组内成员状态。处理:组任务细化、子任务分配到具体
平台(发挥异构性:UAV 空中侦察、UGV 地面抵近/搬运)、组内协同编排、局部重规划。
输出:每个平台的行为序列。

### L3 底层 · 单平台执行器(Qwen2.5-7B + VL-7B,机载边缘,每平台一个)
输入:分配到的行为序列、本机实时感知。处理:语义指令→技能原语(goto/scan/track/…)、
视觉语言理解(目标确认/异常检测)、调用底层控制栈、执行监控与上报。
通信中断时凭本地模型自主降级运行。

## 关键工程要点

1. **任务表示标准化** —— 层间用统一 JSON Schema 传递
   `{group_id, members, objective, constraints, priority, success_criteria, dependencies}`,
   使上层输出能被下层直接解析,并便于校验与可视化。
2. **LLM 输出可靠性** —— 约束解码 / Schema 强制、语法与可行性校验、失败重试与
   规则模板降级;独立的安全监控层对所有指令做几何/物理约束兜底。
3. **事件驱动的闭环重规划** —— 三层皆为「规划—执行—反馈」闭环。底层反馈汇聚成
   组状态触发 L2 重规划;组级失败或新任务才逐级上报触发 L1 重规划,以事件驱动
   避免频繁调用大模型。
4. **人在回路** —— L1 方案生成后由指挥员审核/修改再下发;关键决策需人工确认。
   界面展示推理链以建立信任、便于追责。

## 模型选型逻辑

| 层级 | 模型 | 部署 | 调用频率 | 时延 | 能力需求 |
|------|------|------|---------|------|---------|
| L1 | Qwen3.6-27B | 云/中心站 | 低(分钟级) | 宽松 | 复杂推理、任务分解、全局优化 |
| L2 | Qwen3.6-27B | 边缘/长机 | 中(秒级) | 中等 | 组内协同、动态局部重规划 |
| L3 | Qwen2.5-7B + VL-7B | 机载 | 高(按需) | 严格(<1s) | 指令解析、视觉理解、技能编排 |
